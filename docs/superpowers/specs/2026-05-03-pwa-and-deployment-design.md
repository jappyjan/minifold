# PWA & Deployment Templates — Design Spec

**Date:** 2026-05-03
**Status:** Approved
**Phase:** 9

---

## 1. Goals & Non-Goals

**Goals:**

- Make Minifold installable to the home screen on Android, iOS, and desktop Chromium.
- Brand the installed app per-deployment: PWA name and theme colour come from the operator-set `app_name` and `accent_color` (Phase 8 settings); the PWA icon comes from the operator-uploaded logo when present, otherwise a default Minifold mark.
- Keep the app shell renderable offline so previously-visited directories (Phase 6 IndexedDB cache) remain browsable without network.
- Ship five deployment templates covering the realistic install paths: generic Docker Compose, Coolify, generic Traefik, Unraid Community Applications, and Render one-click deploy.
- Replace the current `docker-compose.yaml` (which is the Coolify variant masquerading as the canonical filename) with a real generic compose file at `docker-compose.yml`.

**Non-Goals:**

- Web push notifications, background sync, periodic sync, File System Access API, or any other "extending PWA" feature from the Next.js PWA guide.
- Custom in-app install banner UI — using `@khmyznikov/pwa-install` web component instead.
- Update-available toast / mid-session SW update UX — `skipWaiting` + `clients.claim` is sufficient; users get the new shell on next navigation after a deploy.
- Per-user PWA preferences (push subscriptions, notification topics, etc.).
- A self-hosted PWA-build pipeline (Bubblewrap / TWA / app-store wrapping). Web installation only.
- Migrating the project off `next/image` for the logo — server-side `sharp` is added but only for icon variants.

## 3. Manifest

### 3.1 `app/manifest.ts`

Dynamic manifest route, exported per Next 16 conventions. Reads settings on every request to reflect operator changes without a restart.

```ts
import type { MetadataRoute } from "next";
import { getDatabase } from "@/server/db";
import { getAllSettings } from "@/server/db/settings";

export const dynamic = "force-dynamic";

export default function manifest(): MetadataRoute.Manifest {
  const settings = getAllSettings(getDatabase());
  const appName = settings.app_name || "Minifold";
  const accent = settings.accent_color || "#3b82f6";

  return {
    name: appName,
    short_name: appName.length > 12 ? appName.slice(0, 12) : appName,
    description: "Self-hosted file browser",
    start_url: "/",
    display: "standalone",
    background_color: "#ffffff",
    theme_color: accent,
    icons: [
      { src: "/api/icon/180/any.png", sizes: "180x180", type: "image/png", purpose: "any" },
      { src: "/api/icon/192/any.png", sizes: "192x192", type: "image/png", purpose: "any" },
      { src: "/api/icon/512/any.png", sizes: "512x512", type: "image/png", purpose: "any" },
      { src: "/api/icon/512/maskable.png", sizes: "512x512", type: "image/png", purpose: "maskable" },
    ],
  };
}
```

`force-dynamic` is required because manifest routes are cached by default; we accept a single SQLite read per manifest fetch (sub-millisecond on WAL) in exchange for live settings reflection.

### 3.2 `app/layout.tsx` head additions

The `<head>` gets four additions (Next 16 App Router renders these from JSX inside the layout's returned `<html>` tree):

- `<link rel="manifest" href="/manifest.webmanifest" />` — Next emits the manifest at this path automatically when `app/manifest.ts` exists.
- `<link rel="apple-touch-icon" href="/api/icon/180/any.png" />` — iOS home-screen icon. iOS ignores the manifest's icon list and only honours this link tag, hence the explicit emission.
- `<meta name="theme-color" content={accent} />` — browser-chrome tint. Settings are already loaded in `RootLayout` (existing pattern).
- An inline early-capture script for `beforeinstallprompt`. The App Router pattern is `<script dangerouslySetInnerHTML>`; it must NOT use `next/script` because that defers, and we need the listener registered before any browser-fired prompt event:

```tsx
<script
  dangerouslySetInnerHTML={{
    __html: `
      window.addEventListener('beforeinstallprompt', (e) => {
        e.preventDefault();
        window.__minifoldInstallEvent = e;
      });
    `,
  }}
/>
```

This must be inline and synchronous so the listener registers before any potential `beforeinstallprompt` event fires — otherwise we miss it on some browsers and `<pwa-install>` falls back to the iOS-style instruction view even on Chromium.

## 4. Icons & Resize Pipeline

### 4.1 Variants

Four icon outputs serve every PWA install:

| Path | Size | Purpose | Consumer |
|---|---|---|---|
| `/api/icon/180/any.png` | 180×180 | `any` | iOS home-screen (`<link rel="apple-touch-icon">`) |
| `/api/icon/192/any.png` | 192×192 | `any` | Android home, browser tab |
| `/api/icon/512/any.png` | 512×512 | `any` | Android splash, desktop install |
| `/api/icon/512/maskable.png` | 512×512 | `maskable` | Android adaptive icons |

The 180×180 size is iOS-specific. iOS doesn't honour the manifest's icon list and instead reads the `<link rel="apple-touch-icon">` we add to the layout (see §3.2). All four variants flow through the same dispatch route (§4.5) — there's no separate `app/apple-icon.tsx`. Maskable is never used at 180 (iOS doesn't support it).

### 4.2 `src/server/settings/icon-rendering.ts`

Pure functions over `sharp`. No I/O, no DB.

```ts
export async function resizeAny(
  input: Buffer,
  size: 180 | 192 | 512,
): Promise<Buffer> {
  return sharp(input)
    .resize(size, size, { fit: "contain", background: { r: 0, g: 0, b: 0, alpha: 0 } })
    .png()
    .toBuffer();
}

export async function composeMaskable(
  input: Buffer,
  accentHex: string,         // "#3b82f6"
): Promise<Buffer> {
  const SIZE = 512;
  const SAFE = Math.round(SIZE * 0.7);  // 70% safe zone per Android maskable spec
  const inset = Math.round((SIZE - SAFE) / 2);

  const logo = await sharp(input)
    .resize(SAFE, SAFE, { fit: "contain", background: { r: 0, g: 0, b: 0, alpha: 0 } })
    .png()
    .toBuffer();

  return sharp({
    create: { width: SIZE, height: SIZE, channels: 4, background: accentHex },
  })
    .composite([{ input: logo, left: inset, top: inset }])
    .png()
    .toBuffer();
}
```

### 4.3 Default icons (`public/icons/`)

`public/icons/icon-source.png` is a 1024×1024 PNG generated via Gemini Nano Banana with this prompt:

> A modern app icon for "Minifold" — a self-hosted file browser for 3D print files (STL, 3MF) and documents. Square 1024×1024 composition. Subject: a single continuous folded paper-ribbon mark forming a stylized lowercase "m" — three connected peaks built from crisp folded planes, like origami. The folds simultaneously suggest a manifold surface (geometry) and stacked file folders. Clean creases, subtle inner shadow where planes meet, soft highlight along front edges. White paper with mild off-white midtones in the shadowed faces. Background: solid medium-blue (#3b82f6) with a very gentle darker radial gradient toward the corners (#2563eb). No noise, no texture, no pattern. Composition: mark is centered and fills the inner 70% of the canvas, with even padding on all four sides — sized for an Android maskable PWA icon safe zone (no detail in the outer 15%, no content near the edges). Style: flat-modern with mild depth, geometric, minimalist. Recognizable at 48 pixels. No text, no lettering, no border, no drop shadow under the mark, no extra decorative elements.

`scripts/build-default-icons.ts` reads this source and writes:

- `public/icons/icon-180.png` — `resizeAny(source, 180)`
- `public/icons/icon-192.png` — `resizeAny(source, 192)`
- `public/icons/icon-512.png` — `resizeAny(source, 512)`
- `public/icons/icon-maskable-512.png` — `composeMaskable(source, "#3b82f6")` (default seeded `accent_color`)

The script runs as `prebuild`. The three derived files are gitignored; the source is committed. Operators who fork and rebrand replace `icon-source.png` and the build regenerates the rest.

### 4.4 Upload flow extension

`src/server/settings/logo-storage.ts` currently writes `/data/logo.<ext>` from operator uploads. Phase 9 extends `saveLogoFromBuffer` (or equivalent) to additionally:

1. Read the uploaded buffer.
2. Run `resizeAny(buf, 180)` → write `/data/logo-180.png` (atomic temp-then-rename).
3. Run `resizeAny(buf, 192)` → write `/data/logo-192.png` (atomic).
4. Run `resizeAny(buf, 512)` → write `/data/logo-512.png` (atomic).
5. Run `composeMaskable(buf, currentAccentHex)` → write `/data/logo-maskable-512.png` (atomic).

A new pure helper `regenerateMaskable(currentLogoBuffer, accentHex)` is exported and called from the accent-colour-change Server Action, so changing accent without changing logo updates only the maskable variant.

If no logo has been uploaded, no `/data/logo-*.png` files exist; the route falls back to the public defaults.

### 4.5 `app/api/icon/[size]/[purpose]/route.ts`

```ts
// GET /api/icon/[size]/[purpose].png
// size: "180" | "192" | "512"
// purpose: "any" | "maskable"
// 1. Validate size + purpose. Allowed combos: 180/any, 192/any, 512/any, 512/maskable. 404 on anything else.
// 2. Determine target file:
//    - 180/any:       /data/logo-180.png       OR public/icons/icon-180.png
//    - 192/any:       /data/logo-192.png       OR public/icons/icon-192.png
//    - 512/any:       /data/logo-512.png       OR public/icons/icon-512.png
//    - 512/maskable:  /data/logo-maskable-512.png OR public/icons/icon-maskable-512.png
// 3. fs.stat the /data candidate; if exists, use it. Otherwise use public.
// 4. Compute ETag from mtimeMs. If If-None-Match matches → 304.
// 5. Stream the file with:
//    Content-Type: image/png
//    Cache-Control: public, max-age=86400, must-revalidate
//    ETag: "<mtimeMs>"
```

The mtime-based ETag means uploading a new logo invalidates browser caches automatically. Public defaults have build-time mtimes; a new deploy → new mtime → fresh fetch.

## 5. Service Worker

### 5.1 Source layout

The SW is authored in TypeScript at `src/sw/sw.ts` and compiled to `public/sw.js` via `scripts/build-sw.ts` (esbuild) at `prebuild` time. This lets us unit-test the cache-strategy logic in Vitest without SW-globals trickery.

### 5.2 Cache strategy

Three named caches:

1. **`shell-v<gitsha>`** — Pre-cached at `install`. Contains:
   - `/` (the app root, RSC stream)
   - `/login`
   - `/_next/static/<chunks>` from a build-time-generated precache manifest list

   Cache-first, never-stale within a deploy. Strategy for navigation requests with no network: serve `/` from this cache so the app shell renders, IndexedDB serves directory listings.

2. **`runtime-static`** — Stale-while-revalidate for:
   - `/_next/image/*`
   - `/api/icon/*` (deployment-public, not user-specific)
   - `/api/logo` (same)

   These have ETag/versioning so SWR is safe, and none of them are auth-gated.

3. **Never cached:**
   - `/api/file/*` (auth-gated file streams)
   - `/api/thumb/*` (auth-gated; access-control resolves per path, so cached thumbs would leak across users with different access)
   - `/api/trpc/*` (auth-gated mutations and queries)
   - `/setup`, `/admin/*` (state-bearing)
   - Any non-GET request
   - Any request with credentialled `Cookie` headers

Pure helper functions live in `src/sw/strategy.ts`:

```ts
export type CacheStrategy = "shell" | "runtime" | "never";

export function getCacheStrategy(url: URL, method: string): CacheStrategy {
  if (method !== "GET") return "never";
  if (url.pathname.startsWith("/api/file")) return "never";
  if (url.pathname.startsWith("/api/trpc")) return "never";
  if (url.pathname.startsWith("/setup")) return "never";
  if (url.pathname.startsWith("/admin")) return "never";
  if (url.pathname.startsWith("/_next/static")) return "shell";
  if (url.pathname === "/" || url.pathname === "/login") return "shell";
  if (url.pathname.startsWith("/api/thumb")) return "never"; // auth-gated; per-path ACL
  if (url.pathname.startsWith("/_next/image")) return "runtime";
  if (url.pathname.startsWith("/api/icon")) return "runtime";
  if (url.pathname === "/api/logo") return "runtime";
  return "never";
}
```

### 5.3 Lifecycle

- **`install`** — Open `shell-v<gitsha>`, `addAll(precacheList)`, call `self.skipWaiting()`. The git SHA is injected at build time into `sw.ts` via an esbuild `define`.
- **`activate`** — Delete all `shell-v*` caches whose name doesn't match the current SHA. Call `self.clients.claim()`. No update-available message — the next navigation picks up the fresh shell.
- **`fetch`** — Dispatch on `getCacheStrategy(...)`:
  - `shell` → cache-first, network fallback. On network error during a navigation request, fall back to the cached `/` (offline shell).
  - `runtime` → stale-while-revalidate against `runtime-static`.
  - `never` → fall through to network; do not touch caches.

Total source ~80 lines including types.

### 5.4 Build-time precache manifest

`scripts/build-sw.ts` runs after `next build` (chained from a `postbuild` script in `package.json`, since it needs the build-manifest to exist). It:

1. Reads `.next/build-manifest.json` to find the root layout chunks (the absolute minimum to render `/`).
2. Emits the chunk paths as a constant array `PRECACHE_LIST` injected via esbuild `define`.
3. Resolves a build identifier in this order: `process.env.BUILD_SHA` (set by CI as a Docker build arg passed via the Dockerfile), then `git rev-parse --short HEAD`, then a build-time ISO timestamp as final fallback. Injects the result as `SHELL_VERSION`.
4. Bundles `src/sw/sw.ts` to `public/sw.js`, format `iife`, target `es2022`, no minification (so deploy diffs are reviewable).

If `.next/build-manifest.json` is missing, the script throws — `next build` must run first. The SW is a build-time artifact; nothing tries to use it in dev (see §6.2).

### 5.5 Headers

`next.config.ts` is extended with explicit headers for `/sw.js`:

```ts
{
  source: "/sw.js",
  headers: [
    { key: "Content-Type", value: "application/javascript; charset=utf-8" },
    { key: "Cache-Control", value: "no-cache, no-store, must-revalidate" },
    { key: "Service-Worker-Allowed", value: "/" },
  ],
}
```

The `no-cache` header is critical: SWs must never be cached by the browser, or new SW versions never roll out.

## 6. Install Prompt

### 6.1 `src/components/pwa/PWAClient.tsx`

Single client component mounted from `app/layout.tsx`. Responsibilities:

1. Register `/sw.js` after window `load`.
2. After 30s, if all conditions hold, mount `<pwa-install>` and call `.showDialog()`:
   - Path is not `/login` nor under `/setup` (`usePathname()` check).
   - Not running standalone (`!window.matchMedia('(display-mode: standalone)').matches`).
   - Not previously dismissed (`!localStorage.getItem("minifold:pwa-dismissed")`).
3. Listen for the component's `pwa-install-success-event` and `pwa-install-user-choice-result-event` events; on dismiss, set `localStorage["minifold:pwa-dismissed"] = "1"`.
4. Pass the saved `window.__minifoldInstallEvent` (captured by the inline script in §3.2) to the component as the `externalprompt-event` attribute, since the component documentation recommends this pattern.

The `@khmyznikov/pwa-install` import is **dynamic** to avoid SSR (`customElements.define()` requires `window`):

```tsx
"use client";
import { useEffect, useState } from "react";

export function PWAClient() {
  const [Lib, setLib] = useState<unknown>(null);
  // ...
  useEffect(() => {
    import("@khmyznikov/pwa-install").then((mod) => setLib(mod));
  }, []);
  // ...
}
```

A type augmentation file `pwa-install.d.ts` declares `<pwa-install>` for JSX so TypeScript stops complaining.

### 6.2 SW registration

```ts
if (
  process.env.NODE_ENV === "production" &&
  "serviceWorker" in navigator
) {
  window.addEventListener("load", () => {
    navigator.serviceWorker
      .register("/sw.js", { scope: "/", updateViaCache: "none" })
      .catch((err) => console.warn("[minifold] SW registration failed:", err));
  });
}
```

The `NODE_ENV === "production"` gate is critical: dev builds do not produce `public/sw.js` (the prebuild script runs only on `next build`), and registering a SW that 404s creates a partially-installed SW state that's hard to clean up. Gating on `production` keeps `next dev` clean.

`updateViaCache: "none"` instructs the browser to bypass HTTP cache when checking for SW updates — belt-and-suspenders alongside the `Cache-Control: no-cache` header on `/sw.js`.

### 6.3 Skip-on-routes

The install prompt does not fire when:

- `pathname.startsWith("/login")`
- `pathname.startsWith("/setup")`
- `pathname === "/change-password"`

All three are pre-auth or forced-state pages where prompting an install is noise.

## 7. Deployment Templates

### 7.1 `docker-compose.yml` (new — canonical generic)

Both services, two named volumes, no labels, no external networks. The drop-in for self-hosters running Caddy / nginx / Tailscale Funnel / their own reverse proxy.

```yaml
services:
  minifold:
    image: ghcr.io/jappyjan/minifold:latest
    pull_policy: always
    restart: unless-stopped
    ports:
      - "3000:3000"
    environment:
      DATABASE_PATH: /app/data/minifold.db
      MINIFOLD_THUMB_SERVICE_URL: http://minifold-thumbs:3001
    volumes:
      - minifold-data:/app/data
      - minifold-files:/files
    healthcheck:
      test: ["CMD", "curl", "-fsS", "-o", "/dev/null", "http://127.0.0.1:3000/"]
      interval: 30s
      timeout: 5s
      retries: 5
      start_period: 15s

  minifold-thumbs:
    image: ghcr.io/jappyjan/minifold-thumbs:latest
    pull_policy: always
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "wget", "--quiet", "--tries=1", "-O", "/dev/null", "http://127.0.0.1:3001/health"]
      interval: 30s
      timeout: 5s
      retries: 5
      start_period: 15s

volumes:
  minifold-data:
  minifold-files:
```

### 7.2 `docker-compose.coolify.yml` (moved — exact current content)

The current `docker-compose.yaml` content is moved here verbatim — Coolify-specific labels, hardcoded resource UUIDs, the long header comment explaining why those labels exist. The `.yaml` extension also changes to `.yml` for consistency.

**Coolify deploy mechanics** (one-time operator step on first deploy of this phase):

1. Merge the PR. Both `docker-compose.yml` and `docker-compose.coolify.yml` are present.
2. In the Coolify UI → Application → General → "Docker Compose Location", change from `docker-compose.yaml` to `docker-compose.coolify.yml`.
3. Click "Deploy". The redeploy uses the new path.

If the path isn't updated, Coolify will deploy from the generic `docker-compose.yml`, which lacks the Traefik labels and will return 503 — the existing operator note about that exact failure mode is preserved in the moved file's header.

### 7.3 `docker-compose.traefik.yml` (new — generic Traefik)

For users running their own Traefik (not Coolify-managed). Uses a single `${MINIFOLD_DOMAIN}` env var instead of Coolify's magic `SERVICE_FQDN_*` substitution.

```yaml
services:
  minifold:
    image: ghcr.io/jappyjan/minifold:latest
    pull_policy: always
    restart: unless-stopped
    networks: [traefik, default]
    environment:
      DATABASE_PATH: /app/data/minifold.db
      MINIFOLD_THUMB_SERVICE_URL: http://minifold-thumbs:3001
    volumes:
      - minifold-data:/app/data
      - minifold-files:/files
    labels:
      traefik.enable: "true"
      traefik.http.routers.minifold.rule: "Host(`${MINIFOLD_DOMAIN}`)"
      traefik.http.routers.minifold.entrypoints: "websecure"
      traefik.http.routers.minifold.tls.certresolver: "letsencrypt"
      traefik.http.services.minifold.loadbalancer.server.port: "3000"
    healthcheck: # same as generic

  minifold-thumbs:
    image: ghcr.io/jappyjan/minifold-thumbs:latest
    # ...same as generic, no Traefik labels (internal-only)...

networks:
  traefik:
    external: true

volumes:
  minifold-data:
  minifold-files:
```

Header comment instructs the operator to set `MINIFOLD_DOMAIN` in `.env` and ensure their Traefik instance has a `letsencrypt` certresolver and an external `traefik` network.

### 7.4 `unraid-template.xml` (new — Unraid CA)

Standard single-container Unraid template (XML schema per Unraid CA conventions). Volume mappings:

- `/mnt/user/appdata/minifold` → `/app/data` (RW; SQLite + uploaded logos)
- `/mnt/user/files` → `/files` (RW; configurable in Unraid UI)

Environment variables exposed:

- `DATABASE_PATH` — defaulted to `/app/data/minifold.db`
- `MINIFOLD_THUMB_SERVICE_URL` — empty default; description explains it enables server-side thumbnails when set

Description body includes: "For server-side thumbnails, also install the `minifold-thumbs` template from Community Applications (search 'minifold thumbs') and set this app's `MINIFOLD_THUMB_SERVICE_URL` to the thumb container's URL."

The icon URL points at a raw GitHub URL for `docs/unraid-icon.png` (a copy of the default 512×512 icon committed for this purpose, since Unraid CA needs a stable external URL).

### 7.5 `render.yaml` (new — Render one-click)

Render permits **one persistent disk per service**. The YAML mounts the SQLite-bearing disk only; the `/files` mount is left to the operator to decide.

```yaml
services:
  - type: web
    name: minifold
    runtime: docker
    repo: https://github.com/jappyjan/minifold
    dockerfilePath: ./Dockerfile
    plan: starter   # cheapest tier with persistent disk support
    healthCheckPath: /
    envVars:
      - key: DATABASE_PATH
        value: /app/data/minifold.db
    disk:
      name: minifold-data
      mountPath: /app/data
      sizeGB: 1
```

**Header comment** explains the two operator paths:

> **Files volume.** Render allows one persistent disk per service. This template mounts only `/app/data` (SQLite + logo uploads). For the file-content volume, choose **one** of:
>
> 1. **S3 storage provider (recommended for Render).** Leave this template as-is; configure an S3-compatible provider in the setup wizard. Phase 3.5 added full S3 support.
> 2. **Single combined disk.** Change `mountPath: /app/data` to a parent like `/data`, then set `DATABASE_PATH=/data/minifold.db`, and bind-mount the file root (e.g. `/data/files`) when configuring the local provider in the setup wizard.
>
> The thumb worker service is not included — add it as a second `web` service if needed (~$7/mo extra on Render's starter tier).

If Render adds multi-disk-per-service before plan-writing time, the YAML adopts the simpler shape with two disks.

## 10. Risks & Open Questions

1. **Render one-disk-per-service.** Asserted from current docs but to be reverified at plan-writing time. If multi-disk is supported, the YAML ships with both `/app/data` and `/files` disks. Falsifiable in 30 seconds; doesn't change architecture.
2. **`<pwa-install>` JSX in React 19 + Next 16.** React 19 has improved custom-element prop handling, but TypeScript will not know the tag. Mitigation: a single-file `pwa-install.d.ts` augmentation declaring the element. Verify hydration works without warnings during plan execution; if it doesn't, fall back to mounting the element imperatively via `ref` in a `<div>` portal.
3. **SW update mid-session.** `skipWaiting` + `clients.claim` means an existing tab navigating after a deploy gets the new shell mid-session — possibly disorienting. Acceptable trade-off vs the no-update-toast simplicity. If users complain post-launch, add a small "App updated, reloading..." toast in a follow-up.
4. **`force-dynamic` manifest cost.** Every install / Lighthouse run / browser-cache-bust hits the DB. SQLite WAL reads are sub-millisecond and the manifest is fetched at most once per browser session, so this is negligible.
5. **Coolify deploy disruption.** The `.yaml`-to-`.yml` rename plus the path change in the Coolify UI is a manual one-time step. The existing file's huge header comment is preserved in the moved file so the rationale travels with the labels. If the operator forgets to update the path, the deploy serves from the generic file and returns 503 (as documented in the moved file's header).
6. **`sharp` binary size.** Adds ~10MB to the production Docker image. Acceptable; the image is already ~400MB with Next + Three.js + the rest of the stack.
